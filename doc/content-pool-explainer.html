③ Fashion Week Categories Odd weeks

A Fashion week also publishes 7 articles using category templates. Unlike QA pools (which use specific questions), Fashion categories define a format and a title pattern — the AI fills in the blanks freshly each week.

Rotation Logic — when more than 7 categories exist

With 8 categories, no single fashion week can use all 8. The planner picks a different 7 each time, rotating through the pool.

④ Full Weekly Flow: Monday → Published Article

Monday 09:00 UTC
Cron fires
POST /api/cron/weekly
node-cron on Fly.io
↓
contentPlanner.server.ts
generateWeeklyPlan()
Checks ISO week number → decides FASHION or QA
↓
If FASHION week
getAllFashionCategories()
  • 1Get 7 built-in + N user-added categories
  • 2If total ≤7: use all 7
  • 3If total >7: rotate — pick starting index from week number
  • 4For each category: GPT fills the title pattern with real specifics
If QA week
getFullQAPool() × 7
  • 1For each of 7 categories: merge base pool + user-added questions
  • 2Filter out already-published topics (deduplication)
  • 3GPT picks 1 question & rewrites it as an SEO article title
  • 4Retry if AI returns a duplicate or placeholder
↓
PostgreSQL via Prisma
blogContentPlan.createMany() — 7 rows
Each row: shop, weekStart, dayIndex, scheduledDate, topic, category, contentFormat, targetWordCount, status="planned"
↓ (each day 10:00 UTC)
Daily 10:00 UTC
Cron fires
POST /api/cron/daily → publishPlanItem()
↓
articleWriter.server.ts — 5 OpenAI calls
Full article pipeline
  • 1enrichKeywords() — generates 8–12 SEO keywords
  • 2researchTopic() — 250–350 word expert brief (GPT-4o)
  • 3generateArticleBody() — 1800–2200 word HTML article (GPT-4o, maxTokens 5500)
  • 4generateArticleMetadata() — title, metaDescription, tags, excerpt (JSON mode)
  • 5publishArticleToShopify() — articleCreate mutation via Admin GraphQL
↓
Result
Article live on encantoshop.es
Row updated: status "planned" → "published", articleId & articleUrl saved. Runway counter decrements by 1.

⑦ Where Data Lives

Source code (hardcoded)

contentPlanner.server.ts

QA_CATEGORIES[] — array of 7 objects, each with:
 • name ("qa-brushes")
 • questionPool: string[] (the base questions)
 • format, titlePattern, targetWordCount

FASHION_CATEGORIES[] — array of 7 objects, each with:
 • name, format, titlePattern, targetWordCount

Database (user-extended)

BlogSettings row (one per shop)

extendedQAQuestions: Json
 {
  "qa-brushes": ["Q1", "Q2"],
  "qa-lashes": [],
  "qa-brow-gel": ["Q1"]
 }

extendedFashionCategories: Json
 [{ name, format, titlePattern, targetWordCount }]
Key design insight: The hardcoded pools are the "seed". User additions are appended from the database on every planning run via getFullQAPool(settings, categoryName). Neither changes the source code — the DB extension is purely additive.
